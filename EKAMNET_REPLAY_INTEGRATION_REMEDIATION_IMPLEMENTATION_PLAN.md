# EKAMNET REPLAY INTEGRITY REMEDIATION IMPLEMENTATION PLAN
## DP / EKAMNET RESEARCH PROGRAM

This document outlines the implementation plan for repairing the three known Replay integrity defects in the active codebase.

---

### 1. Executive Summary

This plan provides a step-by-step remediation strategy for the three database and taxonomy integrity defects identified in the native backtest loop:
1. **Relational lineage ID propagation bug**: Lineage engine outputs are not written back to the Pydantic model before DB persistence.
2. **TheoryStructured ID/timestamp collisions**: Nested sub-objects are duplicated on deepcopy mutations.
3. **Ontology registry mismatch**: `SECTOR_ZSCORE` is omitted from `CORE_CONCEPTS` but generated by the LLM, leading to compliance failures.

The repairs are targeted, non-disruptive to cognitive logic, and lay the groundwork for verifying longitudinal lifecycle closure via Candidate F.

### 5. Defect 1 â€” Relational Lineage Propagation Root Cause

* **Symptom**: Relational SQL tables save a static lineage ID (`7c039bc0-3c76-46d2-bd4b-5692252341e2`) for all theories, while `theory_lineage.json` tracks the correct parent-child lineage.
* **Root Cause Trace**: 
  1. On day 0, a new theory is compiled by `TheoryGenerationFlow.process` (line 557) and initialized with `lineage_id=str(uuid4())`. In one run, this random UUID evaluates to `7c039bc0-3c76-46d2-bd4b-5692252341e2`.
  2. On subsequent days, the novelty gate selects `REVISE` or `REINFORCE` and duplicates the day 0 object using `theory = copy.deepcopy(prior_theory)` (`replay_engine.py:1520` and `1838`).
  3. While `theory.id` is regenerated (`theory.id = str(uuid4())`), the field `theory.lineage_id` is left untouched, inheriting the static ID (`7c039bc0-3c76-46d2-bd4b-5692252341e2`) of the parent.
  4. In `replay_engine.py` line 2018, the lineage engine computes `lineage_id_val` representing the true evolving parent-child lineage node tree.
  5. The divergence point occurs because `lineage_id_val` is returned but never assigned back to `theory.lineage_id`. When `self.theory_repo.save(theory)` is called, the repository saves the static inherited `lineage_id` value, creating a mismatch with `theory_lineage.json`.

### 7. Defect 2 â€” Nested TheoryStructured Identity Root Cause

* **Symptom**: Mutated theories receive a new outer theory ID but duplicate the inner structured ID (`5cb3bc75-f0b9-4883-928e-0736d01c715e`) and timestamp in PostgreSQL.
* **Root Cause**: In `REVISE` and `REINFORCE` novelty paths, `replay_engine.py` mutates the prior theory using `theory = copy.deepcopy(prior_theory)`. It updates the outer ID (`theory.id = str(uuid4())`) but fails to regenerate the nested sub-object's ID (`theory.summary_structured.id`) and timestamp.
* **Identity Semantics Matrix**:

| Operation | Field | Semantic Action | Justification |
| :--- | :--- | :--- | :--- |
| **GENERATE** | `summary_structured.id` | `REGENERATE` | Mints a new unique ID for a fresh proposition. |
| **GENERATE** | `summary_structured.created_at` | `REGENERATE` | Mints the compile-time timestamp. |
| **REVISE** | `summary_structured.id` | `REGENERATE` | Revised theories mutate the core prediction parameters, creating a new distinct proposition. |
| **REVISE** | `summary_structured.created_at` | `REGENERATE` | Current compile-time timestamp of the revision step. |
| **REINFORCE** | `summary_structured.id` | `PRESERVE` | Identifies the same physical proposition being reinforced across days. |
| **REINFORCE** | `summary_structured.created_at` | `PRESERVE` | Preserves the original compilation timestamp of the reinforced belief. |

### 9. Defect 3 â€” Ontology Taxonomy Root Cause

* **Symptom**: The 10-day backtest report lists `0%` valid theory and mechanism components, flagging `SECTOR_ZSCORE` as an unknown value.
* **Root Cause**: The LLM is instructed in the prompt that `sector_zscore` is an available observable variable. The LLM generates components with the concept tag `SECTOR_ZSCORE`. However, `SECTOR_ZSCORE` is omitted from `OntologyRegistry.CORE_CONCEPTS` in `ontology.py`. When `theory_generation_flow.py` runs validation, it rejects the tag as an unknown ontology term, triggering retries and logging compliance failures.
* **Authoritative Source**: `OntologyRegistry` (`cognition/schemas/knowledge/ontology.py`).
* **Cognitive Impact Check**: Does adding `SECTOR_ZSCORE` to `CORE_CONCEPTS` change which generated theories can survive validation?
  - **YES**. When `is_valid` evaluates to `False` due to `SECTOR_ZSCORE` omission, the flow triggers retries and LLM regenerations. Correcting this allows the original generated theory to survive without trigger-retrying.
  - **Classification**: `POTENTIAL_COGNITIVE_EFFECT`. Requires regression testing.

### 11. Cross-Defect Interaction Analysis

- **`LINEAGE PROPAGATION` Ă— `NESTED ID REGENERATION`**: Shared code path inside `replay_engine.py`. Resolving both ensures that the outer theory correctly carries the parent's lineage ID, while its nested structured proposition receives a new unique identifier on mutation.
- **`NESTED ID REGENERATION` Ă— `EVIDENCE FOREIGN KEYS`**: Resolving nested ID collisions ensures that validations and prediction probes in SQL can query nested structured theory fields cleanly without key collisions.
- **Implementation Sequence**:
  1. **Step 1: Baseline Preservation Test Run**: Run the 10-day diagnostic replay on unmodified code to log reference decisions.
  2. **Step 2: Lineage ID Propagation Hotfix (Defect 1) + Preservation Test**: Resolves relational lineage without altering decisions.
  3. **Step 3: Nested ID Regeneration Hotfix (Defect 2) + Preservation Test**: Resolves relational uniqueness of the inner structured object without altering decisions.
  4. **Step 4: Cognitive-Behavior Preservation Confirmation**: Confirm byte-for-byte decision identity for Defects 1+2 in isolation.
  5. **Step 5: Ontology Registry Hotfix (Defect 3)**: Fixed last, since it is a known non-neutral change (`POTENTIAL_COGNITIVE_EFFECT`).
  6. **Step 6: Pre/Post Distribution Comparison**: Run characterization analysis for Defect 3.
  *Rationale for Reordering*: Fixing the non-neutral Ontology registry mismatch (Defect 3) first would confound cognitive-behavior preservation checks for Defects 1 and 2. Keeping Defect 3 last isolates its behavioral impact.
